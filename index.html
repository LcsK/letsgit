What is Git?

  • Distributed version control system
  • Free and open source software
  • Created by Linus Torvalds in 2005

Means unpleasant person in British English slang

Git <> GitHub

You do not need GitHub to use Git.

"GitHub is a web-based Git repository hosting service, which offers all functionality of Git as well as adding its own features."

Getting Started

SSH

Used to connect and authenticate to Git server

  1. Generate SSH Key: ssh-keygen -t rsa -b 4096 -C "your_email@example.com"
  2. Copy the created key from the "id_rsa.pub" file
  3. Add the new key on GitHub: "Settings > SSH and GPG keys > New SSH key"

Repository

A place where the history of your work is stored

  • Local: All changes happens here, until a push occurrence
  • Remote: Refers to the centralized server, where all members will push code
  • All repository's configuration lives in a local ".git" subdirectory
  • Creating a local repository: git init
    Adding a new remote repository: git remote add name git@github.com:username/repo_name

    name is usually defined as origin

  • Creating a local repository from a existing remote repository: git clone git@github.com:username/repo_name

Local workflow

A local repository consists of three "trees" maintained by git

  1. Working Directory: holds the actual files
  2. Index: acts as a staging area
  3. HEAD: points to the last commit you have made

Working with files

  • Verifying all file status: git status
  • Adding files to the staging area: git add file_name git add "*.txt" git add .
  • Removing a file from the staging area: git rm file_name
  • Rollbacking a file in working dir.: git checkout file_name
  • Rollbacking a file in staging area: git reset HEAD file_name

HEAD and Remote

  • Moving file changes from Index to HEAD: git commit -m "Type your commit message here"
  • Pushing your local HEAD to the remote HEAD git push remote_name branch_name

    remote_name is usually defined as origin

Fetching from remote

People could be working cooperatively in the same repository, you should know how to fetch updated data from the remote repository

  • Fetching changes from remote repository: git fetch
    Merging the fetched changes into your local : git merge branch_name
  • The same result of the both commands above can be achieved using: git pull rep_name branch_name

    rep_name is usually defined as origin

Stash

You can store your changes without commiting then

  • Pushing all modifications from Working Dir. to a stack: git stash
  • Listing all stashes stored in the stack: git stash list
  • Applying the last stash stored in the stack: git stash apply

    Or applying by its index on stash list using stash@{0} in the end

  • Removing the stash from de stack: git stash drop
  • You can apply and drop a stash in a single command: git stash pop

Some useful features

  • Listing the commit logs: git log
  • Reseting all files to the last commit: git reset --hard
  • Rollbacking a commit: git reset --hard commit_hash

    The commit hashes can be seen in the git log command

  • Showing differences between the Working Dir. an the Index git diff

Branches

Git's structure is like a tree, the "branch" master is the trunk, and we can have lots of other branches.

Working with branches

  • Creating a branch: git branch branch_name
    Switching to other branch: git checkout branch_name
  • Creating and switching to a new branch in a single command: git checkout -b branch_name
  • Deleting a branch locally: git branch -d branch_name
  • Deleting a branch remotely: git push remote_name --delete branch_name

Best practices

Commit

  • Should be kind of ACID (Atomicity, Consistency, Isolation and Durability)
  • Commit messages should be good, explaining what you have changed and why you have changed
  • Commit messages should be in the imperative, present tense, e.g, "Change", not "changed" or "changes"

Branch

  • A branch should represent a small feature in the code
  • Branch names should be descriptive
  • A branch should be deleted after being merged into the master

And finally...

  • Always fetch the newest version of the code before you push yours.
  • A branch should always have its code reviewed by another dev before merge it into the master, using the Pull Request system
